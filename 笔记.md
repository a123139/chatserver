##### Redis发布-订阅消息队列

**Redis 是一款 高性能、内存级、键值（Key-Value）数据库**

数据默认放**内存**，速度极快.可持久化到硬盘，断电不丢

支持：字符串、哈希、列表、集合、有序集合、**发布订阅**、事务、Lua 等

简单使用

```
1.String 字符串
#存
set name zhangsan
set age 20
# 取
get name
# 自增（计数器）
incr age   # redis本身支持事务处理，多线程对key自增自减是线程安全的
2.Hash 哈希（存对象）
# 单个字段
hset user:100 name lisi
hset user:100 pwd 123456
# 一次性多个字段
hmset user:101 name wangwu age 22
# 取
hget user:100 name
hmget user:101 name age
hgetall user:100
3.List 列表（双向链表）
# 左边加（入队）
lpush msg:list "hello1"
lpush msg:list "hello2"
# 右边取（出队）
rpop msg:list
# 查看范围
lrange msg:list 0 -1
4.Set 集合（无序、去重）
sadd tag:game lol
sadd tag:game dnf
sadd tag:game lol  
# 查看所有
smembers tag:game
# 判断是否存在
sismember tag:game lol
5.ZSet 有序集合（排序 + 去重）
# zadd key 分数 值
zadd rank 95 zhangsan
zadd rank 80 lisi
zadd rank 90 wangwu
# 升序查
zrange rank 0 -1
# 降序排行榜
zrevrange rank 0 -1
6.发布订阅 Pub/Sub
subscribe chat1 #订阅了"zhang san"这个频道，进入订阅阻塞状态，等待该频道信息
publish chat1 "大家好"
7.事务
multi   # 开启事务
set a 1
set b 2
exec    # 提交执行
# 放弃
discard
7.Lua脚本
eval "return redis.call('get','name')" 0
```

**发布/订阅**  

 **subscribe 订阅者**   **publish 发布者**    **Channel 频道**

subscribe(订阅) ------publish(发布)------notify(通知)

✅ **是 典型的观察者模式**

- 观察者模式：

  被观察者（频道 / 发布方）状态变化,自动通知所有**订阅 / 观察者**

- Redis Pub/Sub：

  发布者 = 被观察者,订阅者 = 观察者,有消息立刻全员推送 notify

缺点：无消息持久化，没人订阅时消息直接丢。无堆积、无重试，订阅者掉线，错过的消息拿不到。不可靠消息队列

适合即时通讯、聊天广播、系统通知。（可靠消息队列要用 RabbitMQ、Kafka）

![image-20260426223237038](笔记.assets/image-20260426223237038.png)

此处Redis发布-订阅客户端编程采用的是**同步**的方式**接收消息**(异步用epoll）

![image-20260426223418715](笔记.assets/image-20260426223418715.png)

```
class Redis
{
public:
    Redis();
    ~Redis();

    // 连接redis服务器 
    bool connect();

    // 向redis指定的通道channel发布消息
    bool publish(int channel, string message);

    // 向redis指定的通道subscribe订阅消息
    bool subscribe(int channel);

    // 向redis指定的通道unsubscribe取消订阅消息
    bool unsubscribe(int channel);

    // 在独立线程中接收订阅通道中的消息
    void observer_channel_message();

    // 初始化向业务层上报通道消息的回调对象
    void init_notify_handler(function<void(int, string)> fn);

private:
    // hiredis同步上下文对象，负责publish消息
    redisContext *_publish_context;

    // hiredis同步上下文对象，负责subscribe消息
    redisContext *_subcribe_context;

    // 回调操作，收到订阅的消息，给service层上报
    function<void(int, string)> _notify_message_handler;
};
```

#### 问题更正

##### 1.

client登录后退出，再重新登陆会出现线程阻塞死锁状态

原因：第一版代码中主线程  登录 中有send还有recv

设计时**想**让 主线程 send 子线程 recv，但第一版中主线程1.登录 有send还有recv  登录上后开启了子线程 子线程中有recv，退出登录主线程循环进入菜单页面(这里子线程没有销毁，复用了一直存在，意思是主线程中 a登录主线程中发 子线程中收 a退出登录 循环进入菜单页面 b登录主线程中发 同一个子线程中收 )，但是 在**b再次登录**时 **主**线程执行代码有**send还有recv**   **子**线程也存在**同时**进行**recv**操作 导致**阻塞**

##### 1.更正

初始连接后 就开启子线程（做所有recv操作 聊天/登录ACK/注册ACK）聊天--显示  登录ACK/注册ACK----释放信号量

主线程（所有send操作 登录/注册/退出登录  聊天1v1 /添加好友  创建群组 /加入群组/群聊天）登录/注册-----阻塞等待信号量--服务器应答   聊天1v1 /添加好友  创建群组 /加入群组/群聊天----无需非阻塞循环发送

git config --global user.name "Your Name"

git push origin main  #本地提交好的代码，推送到 GitHub **远程仓库的 main 分支**

#### gdb 调试

基本命令使用

```
info threads #多线程，用这个看所有连接。
thread 2    #切换到某一线程
thread 3
bt   #backtrace缩写  作用：查看当前函数调用栈
```

gdb attach GDB 调试器 挂到**正在运行**的程序上，实时调试

```
ps -ef | grep Chatserve
root      12345  0.0  0.1 123456 7890 ?        Ssl  10:00   0:00 ./Chatserver

gdb attach 12345
```

执行后：

- GDB 会**立刻暂停**这个运行中的程序
- 你可以查看：
  - 堆栈（在哪卡死）
  - 变量值
  - 崩溃位置
  - 线程状态

**普通 gdb 运行程序 = 从零启动，从头调试**

```
c  #continue缩写 作用：attach 后程序是暂停输入c让它继续跑

p userid #打印变量值
p msg
p this->conn

l  #看当前代码位置
q  #退出 GDB 退出后程序会继续正常运行
```



场景 1：程序崩了

```
gdb attach PID
bt  # 直接看崩溃位置
q
```

场景 2：程序卡死 / 不响应

```
gdb attach PID
Ctrl+C
bt      # 看卡在哪
info threads  # 看是不是线程死锁
q
```

场景 3：消息发不出去

```
gdb attach PID
Ctrl+C
bt
p msg  # 查看消息内容是否正确
c
```
